# Project Rules — AI Agent Master Rulebook

## 1. Project Overview
- This document defines the global rules every AI agent (Claude, Copilot, or any automated contributor) must follow when working on this project.
- All code, documentation, and configuration generated by an AI agent must comply with this rulebook without exception.
- If a request conflicts with this document, the agent must flag the conflict instead of silently deviating.

## 2. Architecture
- Follow a layered / Clean Architecture approach: **Presentation → Application (Use Cases) → Domain → Infrastructure**.
- Dependencies point inward only. Domain layer must never import from Infrastructure or Presentation.
- Business logic lives in the Domain/Application layers, never in UI components or route handlers.
- Cross-cutting concerns (logging, auth, caching) are implemented as middleware/services, not duplicated per feature.
- Every module must be independently testable without spinning up the full app.

## 3. Technology Stack
- Document the actual stack here explicitly (language versions, framework, database, hosting) before development starts.
- Do not introduce a new library, framework, or service without updating this section first.
- Prefer stack-native solutions over third-party packages unless justified in writing.

## 4. Folder Structure
```
src/
  domain/           # entities, value objects, business rules
  application/       # use cases, services, DTOs
  infrastructure/     # DB, external APIs, framework adapters
  presentation/       # UI components, controllers, routes
  shared/             # utils, constants, types used across layers
  config/             # environment & app configuration
tests/
  unit/
  integration/
  e2e/
docs/
```
- Feature-based grouping inside each layer is preferred over one giant flat folder.
- No file lives outside its designated layer folder.

## 5. Naming Conventions
- Naming must be descriptive, unambiguous, and consistent across the entire codebase.
- No abbreviations unless industry-standard (e.g., `id`, `url`, `db`).
- Same concept = same name everywhere (no `user` in one file and `usr` in another).

### 5.1 File Naming
- `kebab-case` for files: `user-profile.service.ts`, `order-repository.ts`.
- Suffix files by role: `.service.ts`, `.controller.ts`, `.repository.ts`, `.model.ts`, `.test.ts`.

### 5.2 Class Naming
- `PascalCase`. Suffix by role: `UserService`, `OrderRepository`, `PaymentController`.
- One primary class/export per file.

### 5.3 Function Naming
- `camelCase`, verb-first: `getUserById`, `calculateTotalPrice`, `validateEmail`.
- Boolean-returning functions prefixed with `is`, `has`, `can`, `should`.

### 5.4 Component Naming
- `PascalCase` matching the file name: `UserCard.tsx` exports `UserCard`.
- No generic names like `Component1`, `Wrapper`, `Helper`.

### 5.5 API Naming
- REST endpoints: plural nouns, `kebab-case`: `/api/users`, `/api/order-items`.
- Use HTTP verbs correctly (GET/POST/PUT/PATCH/DELETE); no verbs in the URL (`/getUsers` is forbidden).
- Version APIs explicitly: `/api/v1/...`.

### 5.6 Firestore Naming
- Collections: plural, `snake_case` or `camelCase` — pick one convention and apply project-wide (e.g., `users`, `orderItems`).
- Document IDs: use meaningful IDs when possible; otherwise auto-generated IDs.
- Fields: `camelCase`, consistent types across all documents in a collection.
- No collection may contain documents with divergent schemas — see Schema Drift rules (Section 20).

## 6. Code Style
- One linter/formatter config (ESLint + Prettier or equivalent) is the single source of truth; do not hand-format against it.
- Max function length ~40 lines; extract helpers beyond that.
- No commented-out dead code committed.
- No magic numbers/strings — extract to named constants.

## 7. Documentation Rules
- Every public function/class/module has a short doc comment: purpose, params, return, thrown errors.
- Every folder has a `README.md` if it contains non-obvious structure.
- Docs are updated in the same PR as the code change they describe — never deferred.

## 8. Git Conventions
- One logical change per commit. No "misc fixes" commits.
- Never commit secrets, `.env` files, or credentials.
- Rebase feature branches on latest main before opening a PR; no merge commits inside feature branches.

## 9. Commit Conventions
- Use Conventional Commits: `<type>(<scope>): <description>`.
- Types: `feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `perf`, `style`, `build`, `ci`.
- Example: `feat(auth): add refresh token rotation`.
- Body explains *why*, not just *what*, for anything non-trivial.

## 10. Branch Strategy
- `main` — always deployable.
- `develop` — integration branch (if used).
- `feature/<ticket-id>-<short-desc>`, `fix/<ticket-id>-<short-desc>`, `hotfix/<short-desc>`.
- No direct commits to `main`. All changes via PR with at least one review.

## 11. Security Rules
- Never hardcode secrets, API keys, tokens, or credentials in source code.
- All user input is validated and sanitized server-side, regardless of client-side validation.
- Use parameterized queries / ORM methods — no raw string-concatenated queries.
- Enforce least-privilege access on all database rules, API keys, and service accounts.
- All auth checks happen server-side; never trust client-supplied roles/permissions.

## 12. Environment Variables
- All config values that differ per environment live in environment variables, never hardcoded.
- Provide a `.env.example` with all required keys (no real values).
- Validate required env vars at app startup; fail fast with a clear error if missing.
- Naming: `SCREAMING_SNAKE_CASE`, prefixed by domain when helpful (`DB_HOST`, `STRIPE_SECRET_KEY`).

## 13. Configuration Rules
- All configuration is centralized in `src/config/`, not scattered across files.
- No environment-specific logic (`if (env === 'prod')`) outside the config layer.
- Config objects are typed/validated (schema validation) before use.

## 14. Testing Rules
- Every new feature ships with unit tests for business logic and integration tests for I/O boundaries.
- No PR is merged with failing tests or reduced coverage on touched files.
- Tests must be deterministic — no reliance on real network calls, timers, or random data without seeding/mocking.
- Test file mirrors the source file path under `tests/`.

## 15. Logging Rules
- Use a structured logger (not raw `console.log`) with log levels: `debug`, `info`, `warn`, `error`.
- Never log secrets, tokens, passwords, or full PII payloads.
- Every error log includes enough context (module, operation, correlation/request ID) to trace the failure.

## 16. Error Handling
- No silent `catch` blocks — always log, rethrow, or handle explicitly.
- Use custom error classes/types per domain (`ValidationError`, `NotFoundError`, `AuthError`) instead of generic `Error`.
- User-facing error messages are safe/generic; internal detail stays in logs, not in API responses.
- Fail fast on invalid state; don't let corrupted data propagate silently.

## 17. Performance Guidelines
- Avoid N+1 queries; batch or join where possible.
- Paginate any list endpoint that can return unbounded results.
- Cache expensive/frequently-read, rarely-changed data with clear invalidation rules.
- Profile before optimizing — no speculative micro-optimization that hurts readability.

## 18. Scalability Guidelines
- Design stateless services where possible; no in-memory state that breaks horizontal scaling.
- Long-running or heavy tasks go through background jobs/queues, not synchronous request handlers.
- Database indexes are defined for every query pattern used in production.

## 19. Reusable Code Principles
- Before writing new logic, search the codebase for an existing utility/service that does it.
- Shared logic goes into `shared/` or a dedicated service — never copy-pasted across features.
- Extract a function/module the moment logic is needed in a second place (rule of two).

## 20. DRY Principles
- No duplicated business logic, validation, or constants across files.
- Duplicated code found during review must be refactored into a shared module before merge.
- Configuration values (limits, thresholds, enums) are defined once and imported everywhere.

## 21. SOLID Principles
- **S**ingle Responsibility: one class/module, one reason to change.
- **O**pen/Closed: extend behavior via composition/interfaces, not by editing stable modules.
- **L**iskov Substitution: subtypes/implementations must be swappable without breaking callers.
- **I**nterface Segregation: prefer small, specific interfaces over large general ones.
- **D**ependency Inversion: depend on abstractions (interfaces), not concrete implementations, especially across layers.

## 22. Clean Architecture Principles
- Domain layer has zero framework/library dependencies.
- Use cases orchestrate domain logic; controllers/UI only translate input/output.
- Infrastructure (DB, external APIs) implements interfaces defined by the domain/application layer, not the other way around.
- Swapping a database or framework should never require touching domain logic.

## 23. Code Review Checklist
- [ ] Follows naming conventions (Section 5).
- [ ] No duplicated logic/validation introduced.
- [ ] No hardcoded values/secrets.
- [ ] Tests added/updated and passing.
- [ ] Error handling and logging present at boundaries.
- [ ] Docs updated where behavior changed.
- [ ] No unrelated changes bundled in.
- [ ] Matches architecture/layer boundaries.

## 24. Definition of Done
- Code merged to main branch via reviewed PR.
- Tests written and passing (unit + integration where applicable).
- Documentation updated.
- No new lint/type errors.
- Feature verified against acceptance criteria.
- No known regressions introduced.

## 25. Development Workflow
1. Pick/confirm ticket and acceptance criteria.
2. Create branch per Section 10.
3. Implement following architecture and naming rules.
4. Write/update tests.
5. Update documentation.
6. Self-review against Section 23 checklist.
7. Open PR, address review comments.
8. Merge only after CI passes and approval received.

## 26. AI Collaboration Rules
- AI agents must read this document before generating code for the project.
- AI agents must not invent new architectural patterns, folder structures, or naming schemes outside this document.
- AI agents must ask for clarification (or clearly state assumptions) when a request is ambiguous rather than guessing silently.
- AI agents must not introduce new dependencies without explicit approval.
- AI-generated code must be reviewed by a human before merging to main.
- AI agents must cite which section of this rulebook a non-obvious decision follows, in the PR description.

## 27. Rules Preventing Duplicated Logic
- Before implementing, agent searches codebase for existing implementations of the same behavior.
- Any logic used in 2+ places must be extracted into a shared function/service.
- No duplicate DTOs/types representing the same entity across layers without a documented reason (e.g., mapping boundaries).

## 28. Rules Preventing Duplicated Validation
- Validation rules for an entity are defined once (e.g., in a schema/validator) and reused across API, service, and form layers.
- No re-implementing the same regex/length/format check in multiple files.
- Shared validation schemas are the single source of truth for both client and server where applicable.

## 29. Rules Preventing Schema Drift
- Every entity/collection has one canonical schema definition (type, interface, or schema file).
- Any change to a data shape updates the canonical schema first, then propagates to code/migrations.
- Firestore/DB documents are never written with ad-hoc fields not present in the canonical schema.
- Migrations/versioning are documented when schema changes affect existing data.

## 30. Rules Preventing Inconsistent Naming
- New names must match existing patterns in the same layer/domain before being accepted.
- A single glossary of domain terms is maintained; agents use the exact glossary term, not synonyms.
- Renames are project-wide (all references updated), never partial.

## 31. Rules Preventing Hardcoded Values
- Any value that could change (limits, URLs, timeouts, feature flags, roles) is a named constant or env variable, never inline.
- Magic numbers/strings in conditionals are forbidden — extract to enums/constants.
- Hardcoded values found in review block the PR until extracted.

## 32. Rules for Future Feature Additions
- New features must fit into the existing folder structure and layering (Section 2, 4) — no parallel structures.
- Before adding a feature, check for reusable domain logic/services that already partially solve it.
- New features extend the schema/glossary additively; they do not rename or repurpose existing fields.
- Each new feature includes tests, docs, and a PR description mapping the change to this rulebook.
